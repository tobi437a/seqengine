# seqengine

A C++ engine for the Sequence card game (1v1 variant), with a pybind11
binding and a self-contained Python harness.

## Quick start

```bash
make python      # build the C++ engine + pybind11 module
cd python
python3 play.py  # interactive game; pick [1] for MCTS
```

`make python` needs a C++20 compiler, GNU Make, and `pip install pybind11`.

**Windows:** install GNU Make (e.g. via [MSYS2](https://www.msys2.org/) or
chocolatey's `make`), have `g++` from MinGW-w64 in PATH, then run the
same commands from PowerShell or cmd. The Makefile detects Windows and
swaps `python` for `python3`; shell operations go through Python helpers
in `tools/` so cmd.exe ↔ sh differences don't matter. If `python` isn't
on PATH but `py` is, run `make PY=py python`.

**macOS / Linux:** Xcode CLT or `g++`, plus `pip install pybind11`.

The rest of the project (C++ tests, benchmarks, validation) needs no
Python beyond the stdlib.

## Status

- **Engine:** bitboard state, move generation, sequence detection, dead-
  card mechanics, draw / reshuffle, win detection. Position evaluation
  (window-based Φ and shaping score) is bit-identical to the Python
  reference: 300/300 positions and 1482/1482 shaping scores match within
  1e-15, which is just IEEE round-off.
- **Search:** Single-Observer Information-Set MCTS (Cowling, Powley &
  Whitehouse 2012) with AlphaZero-style PUCT: softmax shaping-score
  priors, lazy expansion with FPU, and tree reuse across moves. One
  tree per worker shared across iterations; each iteration
  re-determinizes the opponent's hand. Tree nodes store moves only,
  not states. The lazy-expansion + tree-reuse search beat the previous
  width-first search by **+30 Elo** (SPRT at 800 iters/move, H1 ≥ 5 Elo
  accepted; 95 % CI +17..+44 over 1,854 games). Against a uniform-
  random opponent it wins ~99 %; against the one-ply heuristic ~75 %
  at 2000 iters (200 games each), which grows with budget.
- **Python bindings:** pybind11 module `_seqengine` with `MCTSConfig`
  and `MCTSEngine` classes. Wrapped by `python/mcts.py`'s
  `MCTSOpponent`, which matches the
  `(game, player, legal_mask) → action_int` signature used by all other
  opponents in this repo.
- **Tests:** all C++ unit tests pass (sequence detection corner cases,
  1000-game random self-play soak, MCTS tactical win, MCTS self-play
  smoke). The MCTS tests run against the current default (ISMCTS).

## Build targets

```
make python      # pybind11 .so into python/ (needs pybind11)
make             # all C++ tools (test, eval_dump, bench, compete, tune, ablation)
make test        # build and run the C++ unit tests
make validate    # cross-check src/eval.hpp against game_engine.py
make bench       # microbenchmark eval + rollouts
make compete     # 200-game match: MCTS vs random and MCTS vs heuristic
make sprt        # SPRT A/B: dev (lazy expansion + tree reuse) vs legacy
                 # (override flags: make sprt SPRT_ARGS="--tc ltc")
make ablation    # n_parallel_trees ablation at fixed budget
make tune        # full hyperparameter grid (slow)
make profile     # play full MCTS games with instrumentation; report
                 # where wall time goes and how often each block fires
                 # per game (override budget: make profile PROFILE_ARGS="10 2000")
make codegen     # regenerate src/board_data.{hpp,cpp} from board_layout.py
make clean       # rm -rf build, rm python/_seqengine*.so
```

## Performance

From `make bench` on an 8-core / 16-thread dev box (16 threads, `-O3`):

| Workload                       | Throughput          |
|--------------------------------|---------------------|
| `total_phi` (192-window eval)  | ~19.7M calls/s      |
| Random rollouts (full games)   | ~1.1M games/s       |
| Heuristic decision             | ~4.9M decisions/s   |
| MCTS iterations (game-avg)     | ~259.2k iters/s     |

The MCTS row averages 1000-iter searches over 20 sampled mid-game
positions. The initial position alone is ~2× slower because rollouts from an empty board are
~2× longer.

Note the MCTS row is ~13% slower per iteration than the legacy
(pre-lazy-expansion) search: selection now computes shaping-score
priors at every descent step instead of only at fully-expanded nodes.
Each iteration buys more in exchange — deeper descents through reusable
subtrees instead of width-first sweeps — and the trade nets out well
ahead (SPRT: +30 Elo at a fixed iteration budget, see *MCTS design*).

Reading this:

- **Frequency × cost is the signal.** `state::make_move` is fast per
  call but fires ~2 M times per game, putting it at 11 % of
  wall — optimizing one 0.1 % function is rarely worth it; this view
  makes that obvious.
- **The rollout policy is the bottleneck.** `pick_rollout_move` is 62 %
  of wall. That's the lever for any future "make MCTS faster" work —
  a value-network rollout would replace exactly this block.
- **Selection got more expensive by design.** `MCTS::select_puct` rose
  from ~2.5 % of wall (legacy) to ~11 %: lazy expansion runs prior-
  weighted PUCT at every descent step instead of only at fully-expanded
  nodes, and descents go deeper. That cost is what bought the +30 Elo
  — see *MCTS design*.

## Using the engine from Python

```python
from game_engine import SequenceGame
from seq_actions import get_legal_action_mask, int_to_action
from mcts import MCTSOpponent

mcts = MCTSOpponent(iterations=1000, n_parallel_trees=8)

game = SequenceGame()
game.reset()

while not game.done:
    mask = get_legal_action_mask(game, game.current_player)
    action_int = mcts(game, game.current_player, mask)
    game.step(int_to_action(action_int))
```

`MCTSOpponent.__call__` accepts the same `(game, player, legal_mask)`
signature as `heuristic_action` and `random_action`, so it drops straight
in as a swappable opponent. The legal-mask is ignored (MCTS re-derives
legal moves from its own bitboard state) but is part of the signature for
compatibility with the rest of the harness.

## MCTS design

Single-Observer Information-Set MCTS (Cowling, Powley & Whitehouse
2012). Public-info perfect-information MCTS, made hidden-info-aware via
per-iteration determinization:

1. **Per-iteration determinization.** ONE tree shared across all
   iterations of a worker; every iteration freshly re-determinizes the
   opponent's hand and deck order, then walks the tree using only
   children whose move is legal in the current determinization. Tree
   nodes store *moves*, not states; the live state is rebuilt by
   replaying moves from the determinized root each iteration. The cost
   is one `make_move` per descent step; the benefit is that a single
   search budget sees as many distinct guesses of the opponent's hand
   as it has iterations. This directly addresses *strategy fusion* —
   the failure mode where the search commits to a plan optimized for
   one (wrong) guess of the hidden information.

2. **PUCT selection with shaping priors.** Selection maximizes
   Q(a) + c · P(a) · √(Σ_b N(b) + 1) / (1 + N(a)), where the prior
   P(a) is a softmax over each legal move's shaping score, recomputed
   per descent step (hands diverge across determinizations). The visit
   sum runs over the *currently-legal* children only, so moves legal in
   few determinizations aren't unfairly punished — the concern classic
   ISMCTS handles with per-child availability counts, folded into PUCT
   here. Ties are broken uniformly at random — without that, structural
   ties (e.g. dead-card declarations at winrate ≈ 1.0) accumulate
   visits by move-generation order. The random tiebreak alone was the
   biggest contributor to playing strength once the algorithm was
   correct.

3. **Robust-child final selection.** Argmax visits over root children
   (not argmax mean). Visit count is the more stable signal because it
   incorporates the bandit's own confidence in its estimates.

4. **ε-greedy heuristic rollouts.** Rollout policy is greedy on
   `shaping_score` with probability 1−ε (default ε=0.30); ε of the time
   it picks uniformly. Pure greedy makes every rollout from a state
   identical and defeats MCTS.

5. **Two RNGs.** One inside `GameState` for game-inherent randomness
   (initial shuffle, rare reshuffle when the deck empties), one inside
   `MCTSEngine` for search (determinizations + rollout choice). Keeps
   rollouts from being correlated through the game RNG.

6. **Length-decay knob.** `cfg.length_decay` (default 1.0 = off) lets
   you prefer fast wins / slow losses by setting it `< 1`. An earlier
   ablation showed no overall winrate benefit, but it's useful for
   tactical sanity tests where multiple winning continuations tie at
   +1.0.

7. **Lazy expansion with FPU.** (`cfg.lazy_expansion`, default ON.)
   Selection runs PUCT over *all* legal moves at a node: visited
   children use their empirical Q, unvisited moves use a first-play-
   urgency value of parent-Q − `cfg.fpu_reduction` (default 0.10), and
   a node is only materialized when actually selected. The legacy mode
   (OFF) force-expanded every untried legal move before PUCT could kick
   in — at ~50 legal moves per node that burned most of a small
   per-tree budget sweeping width-first. SPRT (lazy+reuse vs legacy,
   800 iters, 927 pairs): **+30 Elo, 95% CI +17..+44**.

8. **Tree reuse across moves.** (`cfg.tree_reuse`, default ON.) The
   engine keeps its root-parallel forest between `suggest_move` calls.
   The harness reports each actually-played move via
   `MCTSEngine::advance(move)`; the next search re-roots every tree at
   the node matching the played path, carrying over subtree statistics.
   Any tree where the path has no node — and all trees when `advance`
   was never called — resets to a fresh root, so harnesses that don't
   call `advance` (e.g. the Python binding today) get exactly the old
   search-from-scratch behavior.

9. **Root-parallel trees.** `cfg.n_parallel_trees` (default 8) builds
   N independent ISMCTS trees and aggregates their root-level visit
   counts. Because each tree already re-determinizes per iteration,
   the extra trees don't add determinization diversity — they're a
   pure parallelism knob. Trees are dispatched to `cfg.n_threads`
   workers via `std::async`; scaling is near-linear up to
   `min(n_threads, n_parallel_trees)` cores. The Python wrapper
   defaults `n_threads=0` (auto via `hardware_concurrency()`, capped
   at `min(n_parallel_trees, 16)`); the C++ default is `n_threads=1` so
   tests, benchmarks, and the compete harness stay deterministic.

### What was tried and rejected

- **γ-discounted sum of step rewards as rollout value** — variance
  dominated signal. The dense reward signal sounds right in principle
  but was strictly worse than terminal ±1 in head-to-head.
- **Partial-credit terminal value** (terminal + α · Δseqs at end of
  rollout) — similar story; marginal at best.
- **Sequence-completion override in rollout policy** — the most
  counter-intuitive result. Forcing both rollout players to grab obvious
  sequences collapses rollouts into deterministic tempo races,
  amplifying tiny differences. Kept as a config flag
  (`prefer_completing_moves`, default OFF) but documented as harmful.

The pattern, repeated three times: in MCTS, lower-variance value
estimates beat denser-but-noisier ones.

## Other design notes

### Bitboards

100 cells don't fit in one `uint64_t`, so `BB100` is a pair: `lo` (cells
0..63) and `hi` (cells 64..99 with the top 28 bits masked off). Standard
ops: `|`, `&`, `~`, `popcount`, `test/set/reset`, plus `pop_lsb` for
iterating set bits.

Per-player state lives in two pairs of bitboards:

- `chips[2]`  — unlocked chips per player
- `locked[2]` — chips claimed by a completed sequence (immune to
  one-eyed jacks; can be re-used by a second overlapping sequence
  via the "share ≤ 1 chip" rule)

Joker corners aren't stored in any of these — they live in the static
`JOKER_MASK` and count as "everyone's chip" for sequence purposes.

### Sequence detection

The 192 possible 5-in-a-row windows are precomputed as `BB100` masks, and
each cell has a precomputed list of the ≤20 window indices passing
through it. When a chip is placed at cell `c`, sequence detection only
inspects those windows. Iteration order (direction-major, then position)
matches `game_engine.py`'s `_check_sequences` exactly, because earlier
windows in a single call can lock chips that disqualify later candidate
windows under the share-≤1 rule.

### Codegen

Board layout (suit assignments, joker corners, card→cells map) lives in
`python/board_layout.py` and isn't worth maintaining by hand on the C++
side. `tools/gen_board_data.py` reads it and emits the C++ tables. Run
`make codegen` after editing the board.
